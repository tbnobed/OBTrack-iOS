=== FIX 1 (P0) — Packet timestamp uses the wrong clock ===
File: TrackingPacket.swift and ARTrackingManager.swift
Problem: TrackingPacket.from(...) sets `timestamp: Date().timeIntervalSince1970`
(wall clock), but the send throttle in ARTrackingManager uses `frame.timestamp`
(ARKit's monotonic capture time). The transmitted timestamp must be the ARKit
capture time, not wall clock.
Change:
- In TrackingPacket.swift, change the factory signature to:
    static func from(camera: ARCamera, frame: Int, timestamp: TimeInterval) -> TrackingPacket
  and set the packet's `timestamp` field to that parameter (delete the Date() call).
- In ARTrackingManager.swift, where it builds the packet (the rate-limited send
  block, currently `TrackingPacket.from(camera: camera, frame: currentFrame)`),
  pass the existing `now` value (which is `frame.timestamp`) as `timestamp: now`.
Acceptance: the packet's timestamp equals frame.timestamp; no Date() remains in
TrackingPacket.swift.

=== FIX 2 (P0) — Rotation is not converted to Unreal's coordinate frame ===
File: freed_bridge.py
Problem: remap_position() converts position from ARKit to Unreal axes
(includes the right-handed -> left-handed flip), but quat_to_euler_deg()
decomposes the quaternion in raw ARKit axes with no matching basis change.
Position and rotation therefore live in different coordinate frames, so combined
moves look mirrored / rotated about the wrong axis in Unreal.
Change — make position AND rotation share ONE basis so they can never diverge:
- Define the change-of-basis matrix once, matching the existing position remap
  (UE_x,UE_y,UE_z) = (-z, x, y):
      M = [[0,0,-1],
           [1,0,0],
           [0,1,0]]
- Add plain-Python helpers: mat3_mul(A,B), mat3_transpose(A), mat3_vec(A,v),
  and quat_to_R(qx,qy,qz,qw) returning the standard 3x3 rotation matrix.
- Rewrite the conversion so:
    * position: (ue_x,ue_y,ue_z) = mat3_vec(M, (x,y,z))   # replaces remap_position
    * rotation: R_ue = mat3_mul(mat3_mul(M, quat_to_R(...)), mat3_transpose(M))
      then extract yaw/pitch/roll from R_ue.
- IMPORTANT: the exact Euler extraction order/signs that Unreal's FreeD rotator
  expects must be verified empirically in Unreal — do NOT claim it is verified.
  Put the extraction in a small function with the order clearly documented, and
  expose three sign multipliers and an order constant at the top of the file
  (e.g. EULER_ORDER = "ZYX", YAW_SIGN/PITCH_SIGN/ROLL_SIGN = 1) so the human can
  flip them during on-set calibration without touching the math.
- Keep feeding the resulting pan/tilt/roll and x/y/z into build_freed_packet()
  exactly as before (units unchanged: degrees and meters).
Acceptance: position and rotation are both derived from the single matrix M;
the FreeD packet bytes/scaling/checksum are unchanged; the sign/order knobs exist
and are commented as requiring empirical verification.

=== FIX 3 (P1) — ARSession delegate runs on the main thread ===
File: ARTrackingManager.swift
Problem: session.delegateQueue is never set, so the per-frame delegate work runs
on the main thread and competes with SwiftUI. Also `let currentFrame =
frameCount + 1` reads the @Published counter on the delegate thread, which
becomes a data race once the delegate moves off main.
Change:
- Add a dedicated serial queue (e.g. `private let sessionQueue =
  DispatchQueue(label: "com.obtrack.arsession")`) and set
  `session.delegateQueue = sessionQueue` in init.
- Replace the frame counter: add `private var sendSeq: Int = 0`, increment it on
  the session queue, use it as the packet `frame`, and publish a copy to the
  @Published `frameCount` via DispatchQueue.main.async.
- Ensure EVERY @Published mutation (trackingState, position, rotation,
  frameCount, udpStatus, depthImage) is written on the main queue. Keep the UDP
  send and depth dispatch off the main queue.
Acceptance: no @Published property is mutated off the main thread; no shared
mutable state is read on one queue and written on another.

=== FIX 4 (P1) — UDPClient.sendStatus cross-thread race ===
File: UDPClient.swift and ARTrackingManager.swift
Problem: sendStatus is marked @Published on a class that is not an
ObservableObject, is written on the UDP queue, and is read on the AR thread.
Change:
- Remove @Published from sendStatus; make it `private(set) var sendStatus`
  updated only on the UDP queue.
- Add `var onStatusChange: ((String) -> Void)?` to UDPClient, invoked whenever
  sendStatus changes.
- In ARTrackingManager, set udpClient.onStatusChange to marshal the value to the
  @Published `udpStatus` on the main queue, and delete the direct
  `self.udpClient.sendStatus` read in the send block.
Acceptance: UDPClient status is never read directly across threads.

=== FIX 5 (P1) — Add a "Lite (shooting)" mode ===
Files: ARTrackingManager.swift, ContentView.swift, ARCameraView.swift
Problem: scene mesh reconstruction + depth semantics + the depth heatmap are
heavy sustained load on a device whose only job is clean pose; this risks
thermal throttling during long takes. None of it is needed for tracking.
Change:
- Add a `liteMode: Bool` parameter to startTracking (default false). When true:
  do NOT set sceneReconstruction, do NOT insert depth frameSemantics, and skip
  all depth-image processing (leave depthImage nil).
- In ContentView, add a "Lite (shooting) mode" toggle in the settings panel
  (disabled while tracking, like the other settings), and pass it into
  startTracking. When lite mode is on, the depth/mesh visual toggles should be
  hidden or disabled.
- In ARCameraView, when there is no depth image / mesh, simply show the camera
  feed without those overlays (it already handles nil depthImage).
Acceptance: with Lite mode on, the session runs world tracking only (no mesh, no
depth, no heatmap) and still streams pose over UDP.
